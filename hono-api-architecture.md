# Hono API Architecture

## Overview

This API is a Cloudflare Worker built with Hono, `@hono/zod-openapi`, Better Auth, Drizzle, PostgreSQL/Neon, and R2. The codebase is organized by boundaries:

- `src/index.ts` owns app-wide wiring.
- `src/modules/{domain}` owns HTTP contracts, route guards, handlers, and schemas.
- `src/services` owns business logic.
- `src/repositories` owns database access.
- `src/db/schema` owns table definitions.
- `src/lib`, `src/utils`, `src/middleware`, `src/types`, and `src/constants` own shared app primitives.

Keep the flow predictable:

```txt
request
  -> src/index.ts global middleware
  -> module route guard
  -> OpenAPI/Zod validation
  -> handler
  -> service
  -> repository
  -> database / external binding

errors
  -> errorHandler
  -> { error: { code, message, details } }
```

## Directory Rules Of Thumb

### `src/index.ts`

- Create `new OpenAPIHono<AppEnv>()`.
- Register `errorHandler`, CORS, `/openapi.json`, `/docs`, logger, `dbMiddleware()`, `withAuth`, root health text, and mounted modules.
- Put only app-wide middleware here. Domain-specific auth/role rules belong in module route files.

### `src/modules/{domain}`

- `*.route.ts`: define `createRoute(...)` contracts, OpenAPI response schemas, route guards, and `OpenAPIHono<AppEnv>()` wiring.
- `*.handler.ts`: read validated input, read context values, build a service, call one service method, and return the declared response.
- `*.schema.ts`: define request, params, query, and response schemas. Prefer shared helpers from `src/lib`.
- Do not put business rules or raw Drizzle queries in module files.

### `src/services`

- Put business logic here.
- Services are factory functions such as `createCoursesService(coursesRepo)`.
- Inject repositories, or multiple repositories when one workflow spans resources.
- Throw `Errors.*` for not found, conflict, validation, unauthorized, forbidden, and internal failures.
- Do not import Hono `Context` or construct HTTP responses.

### `src/repositories`

- Put Drizzle queries here.
- Repositories are factory functions such as `createCoursesRepo(db)`.
- Return rows, totals, cursors, created/updated/deleted records, or `null` for missing single records.
- Own search, sort, relation loading, pagination query mechanics, `.returning()`, and timestamp updates.
- Do not throw HTTP/domain errors unless the failure is truly persistence-specific.

### `src/db`

- `src/db/index.ts` creates Drizzle clients and handles local `pg` versus Neon HTTP behavior.
- `src/db/schema/*.ts` defines tables and inferred select/insert types.
- `src/db/schema/index.ts` re-exports schema files.
- `src/db/migrations` is generated by Drizzle Kit. Treat migration SQL and metadata as generated unless intentionally changing schema.

### `src/middleware`

- Middleware reads/writes Hono context or enforces access.
- If middleware sets `c.set('name', value)`, add the variable to `AppVariables` in `src/types/app.ts`.
- `withAuth` hydrates auth/session context. It does not protect routes by itself.
- `requireAuth` and `requireRole(...)` are route-level guards.

### `src/lib`

- Put shared app primitives here: auth factory, typed errors, OpenAPI helpers, common schemas, pagination helpers, auth header forwarding, and table-to-Zod helpers.
- Helpers here should be reusable across modules and not depend on one domain.

### `src/utils`

- Put small standalone utilities here, such as file validation and crypto helpers.
- If a helper becomes domain-specific business logic, move it to the matching service.

### `src/types` And `src/constants`

- `src/types/app.ts` is the Hono environment contract for bindings and variables.
- Keep Cloudflare binding names synchronized with Worker generated types.
- Roles and error codes are shared constants; update all rank maps and type definitions together.

## One Complete Example

The `courses` module covers the common API use cases: public reads, protected writes, params/query/body validation, offset pagination, cursor pagination, create, update, delete, not-found errors, and repository-backed Drizzle access.

```txt
src/modules/courses/courses.schema.ts
  -> derives select/insert/update schemas from the Drizzle table
  -> reuses UuidIdParamsSchema, OffsetQuerySchema, CursorQuerySchema
  -> exports response schemas with createOffsetResponseSchema/createCursorResponseSchema

src/modules/courses/courses.route.ts
  -> defines list, cursor list, get, create, update, delete createRoute(...) contracts
  -> uses jsonOk/jsonCreated/deletedNoContent and shared error responses
  -> wires public reads first
  -> applies coursesRoute.use('/*', requireAuth, requireRole('personnel')) before writes

src/modules/courses/courses.handler.ts
  -> builds createCoursesService(createCoursesRepo(c.get('db')))
  -> reads c.req.valid('query'), c.req.valid('param'), or c.req.valid('json')
  -> wraps lists with createOffsetPage(...) or createCursorPage(...)
  -> returns c.json(payload, status) or c.body(null, 204)

src/services/courses.service.ts
  -> accepts CoursesRepo
  -> keeps business decisions such as field picking and not-found behavior
  -> throws Errors.notFound('Course not found') when repo returns null

src/repositories/courses.repo.ts
  -> accepts Db
  -> runs Drizzle select/insert/update/delete queries
  -> uses .returning() for mutations
  -> updates updatedAt during update
  -> implements search, sort, count, offset pagination, and cursor pagination
```

The same shape works for most resource modules. If a workflow spans resources, inject multiple repositories into one service, as the applications service does for applications, parents, and addresses.

### Compact Handler/Service/Repository Shape

```ts
const getCoursesService = (c: Context<AppEnv>) =>
	createCoursesService(createCoursesRepo(c.get('db')));

export const createCourse: RouteHandler<typeof createCourseRoute, AppEnv> =
	async (c) => {
		const body = c.req.valid('json');
		const course = await getCoursesService(c).create(body);
		return c.json(course, 201);
	};

export const createCoursesService = (coursesRepo: CoursesRepo) => ({
	async update(id: string, data: Partial<NewCourse>) {
		const course = await coursesRepo.update(id, data);
		if (!course) throw Errors.notFound('Course not found');
		return course;
	},
});

export const createCoursesRepo = (db: Db) => ({
	update: (id: string, data: Partial<NewCourse>) =>
		db.update(courses)
			.set({ ...data, updatedAt: new Date().toISOString() })
			.where(eq(courses.id, id))
			.returning()
			.then((r) => r[0] ?? null),
});
```

### Multipart / R2 Exception

File upload handlers are the main exception to JSON validation:

```ts
const body = await c.req.parseBody();
const uploadedFile = fileValidator(body.file);

await c.env.R2.put(key, uploadedFile.stream(), {
	httpMetadata: { contentType: uploadedFile.type },
});

return c.json({ url: `${c.env.PUBLIC_URL}/${key}` }, 201);
```

Keep validation shared in `src/utils` when it is reusable. Keep storage binding names in `src/types/app.ts` synchronized with generated Worker types.

## Response And Error Rules

- Successful JSON responses are direct and route-specific.
- Paginated responses use `createOffsetPage(...)` or `createCursorPage(...)`.
- Deletes return `c.body(null, 204)` when the route declares no content.
- Do not introduce a universal response envelope unless every affected route schema and handler changes together.
- Known failures should be thrown as `Errors.*` or `HTTPException` where the existing boundary uses it.
- `errorHandler` returns:

```json
{
	"error": {
		"code": "NOT_FOUND",
		"message": "Resource not found",
		"details": []
	}
}
```

## Adding A New Domain

1. Add or update the Drizzle table in `src/db/schema/{domain}.ts`.
2. Export it from `src/db/schema/index.ts`.
3. Add `src/repositories/{domain}.repo.ts`.
4. Add `src/services/{domain}.service.ts`.
5. Add `src/modules/{domain}/{domain}.schema.ts`.
6. Add `src/modules/{domain}/{domain}.handler.ts`.
7. Add `src/modules/{domain}/{domain}.route.ts`.
8. Mount the route in `src/index.ts`.
9. Generate migrations when schema changes.
10. Run `bun run typecheck` when dependencies are available.

The main rule: HTTP contracts live in modules, decisions live in services, persistence lives in repositories, shared infrastructure lives in `lib`/`middleware`, and generated or runtime binding files stay synchronized with configuration.
